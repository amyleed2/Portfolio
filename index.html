About Me

How I Work

운영 중인 앱을 안정적으로 유지하고, 필요한 변화는 점진적으로 구조에 녹여내는 iOS 개발자입니다.

커머스, 콘텐츠, 플랫폼 도메인에서 모바일 앱을 개발하고 운영하며 안정성, 유지보수성, 협업 비용을 함께 고려해왔습니다. 첫 회사인 웹에이전시에서 여러 SI 프로젝트를 수행하며 디자이너, 기획자, 다양한 파트의 개발자와 협업하는 방식을 익혔고, Android 기반 초기 경험은 이후 iOS 개발과 운영 구조를 이해하는 기반이 되었습니다.

01

문제를 재현 가능한 단위로 좁힙니다

운영 안정성을 판단하는 방식

운영 중인 앱에서 이슈가 발생하면 먼저 재현 조건, 영향 범위, 배포 리스크를 분리해 봅니다. Crashlytics, API 로깅, VOC를 단순 확인 자료가 아니라 원인을 좁히는 단서로 사용하고, 사용자 영향이 큰 문제부터 개선합니다.

02

한 번에 갈아엎기보다 안전한 변경 단위를 찾습니다

레거시를 다루는 방식

레거시 구조를 볼 때는 먼저 현재 코드가 유지되어 온 이유와 의존성을 확인합니다. UIKit 기반 화면도 전체 교체보다 영향 범위를 나누고, 필요한 지점부터 SwiftUI, MVVM 등으로 옮겨갈 수 있는 변경 단위를 찾습니다.

03

기술보다 먼저 적용 맥락을 봅니다

기술을 선택하는 기준

UIKit, SwiftUI, RxSwift, Combine은 각각 장점이 있지만 모든 상황의 정답은 아니라고 봅니다. 기존 코드와 팀의 운영 방식, 테스트 가능성, 이후 유지보수 비용을 함께 보고 어디까지 적용할지 결정합니다.

04

반복되는 논의를 구조로 줄입니다

협업과 공통화를 설계하는 방식

웹에이전시와 SI 프로젝트를 거치며 디자이너, 기획자, 서버/웹/모바일 개발자와 요구사항을 맞추는 협업을 꾸준히 해왔습니다. 협업 비용은 좋은 의지만으로 줄어들지 않기 때문에, 디자인 시스템과 공통 컴포넌트처럼 반복되는 구현과 논의를 구조로 만들고 사용 기준과 예외 케이스를 문서와 예시로 남기는 방식을 선호합니다.

05

도구는 활용하되 판단은 직접 검증합니다

AI를 사용하는 기준

Cursor, Claude Code, Copilot 같은 AI 도구는 레거시 분석, 설계 검토, 테스트 케이스 작성의 속도를 높이는 데 활용합니다. 다만 AI가 제안한 구조를 그대로 받아들이기보다 기존 코드와 운영 리스크에 맞는지 검증하고, 최종 설계 판단과 품질 책임은 개발자가 가져가야 한다고 봅니다.
