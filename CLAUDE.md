# Project purpose
이 프로젝트는 iOS 개발자 이지은의 정적 포트폴리오 웹사이트다.
목표는 채용 담당자, 실무 면접관, 테크 리드 관점에서 포트폴리오의 전달력과 설득력을 높이는 것이다.

# Project structure overview
이 프로젝트는 HTML, CSS, Vanilla JavaScript 기반의 정적 사이트다.

주요 파일 역할:
- index.html: 섹션 구조, 메인 텍스트, 전체 레이아웃
- styles.css: 전체 스타일, 반응형 레이아웃, UI 디자인
- script.js: 네비게이션, 스크롤, 폼 처리, EmailJS 관련 동작
- slider.js: 프로젝트 슬라이더 동작
- career-modal.js: 경력 상세 모달 데이터 및 렌더링
- project-modal.js: 프로젝트 상세 모달 데이터 및 렌더링
- assets/: 이미지 및 정적 리소스
- EMAILJS_SETUP.md: EmailJS 설정 참고 문서
- README.md: 프로젝트 사용 및 실행 문서

# Working rules
- 항상 먼저 관련 파일을 읽고, 무엇을 왜 수정할지 짧게 설명한 뒤 수정한다.
- 사용자가 요청하지 않으면 전체 구조를 불필요하게 갈아엎지 않는다.
- 사용자가 요청하지 않으면 파일명, 폴더명, 이미지 경로를 바꾸지 않는다.
- 사용자가 요청하지 않으면 assets/ 내부 파일은 삭제하거나 이동하지 않는다.
- 사용자가 요청하지 않으면 PDF 파일은 수정하지 않는다.
- 경력/프로젝트 문구 수정은 기존 데이터 구조를 유지한 채 진행한다.
- 텍스트를 개선할 때는 역할, 문제, 해결 방식, 결과가 드러나게 쓴다.
- 근거 없는 성과 수치, 허위 성과, 과장 표현은 새로 만들지 않는다.
- 큰 수정 전에는 가능하면 수정 초안을 먼저 제안하고, 그 다음 실제 파일을 수정한다.
- HTML 문자열이 포함된 JavaScript 파일을 수정할 때는 따옴표, 백틱, 템플릿 문자열, DOM 구조가 깨지지 않게 주의한다.

# Review lens
항상 아래 3가지 관점으로 본다.

1. 채용 담당자 관점
- 첫인상이 명확한지
- 경력 요약이 빠르게 읽히는지
- 강점이 선명하게 보이는지
- 문장이 장황하지 않은지

2. 실무 면접관 관점
- 프로젝트 설명에 기술적 깊이가 있는지
- 문제 해결 방식과 판단이 드러나는지
- 단순 기술 나열에 그치지 않는지
- 실제로 질문하고 싶은 포인트가 보이는지

3. 테크 리드 관점
- 설계 판단과 유지보수성이 보이는지
- 협업 맥락이 드러나는지
- 우선순위 판단과 기여도가 보이는지
- 시니어 개발자로서 설득력이 있는지

# Editing priorities
포트폴리오 개선 요청 시 아래 우선순위를 따른다.

1. 전달력 개선
- 첫 문장
- 경력 요약
- 프로젝트 핵심 소개

2. 문장 개선
- 모호한 표현 제거
- 기술 나열형 문장 개선
- 성과/기여 표현 강화

3. 구조 개선
- 섹션 순서
- 정보 밀도
- 가독성

4. 스타일 개선
- spacing
- hierarchy
- responsive layout

# EmailJS caution
script.js에는 EmailJS 관련 placeholder 값이 있을 수 있다.
- 사용자가 요청하지 않으면 service key, template id, public key 관련 로직을 임의로 확정하지 않는다.
- 실제 키 값은 만들어내지 않는다.
- EmailJS 관련 수정 시 기존 동작을 깨지 않도록 보수적으로 수정한다.

# Quality checklist
수정 후 아래 항목을 확인한다.
- 메인 페이지가 정상적으로 열린다.
- 네비게이션 이동이 동작한다.
- Career 모달이 정상적으로 열린다.
- Project 모달이 정상적으로 열린다.
- 이미지 경로가 깨지지 않는다.
- 모바일 화면에서 레이아웃이 심하게 깨지지 않는다.
- JavaScript 에러가 새로 생기지 않는다.
